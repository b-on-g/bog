# Что такое $mol

$mol – это реактивный микромодульный фреймворк для фронтенда, разработанный Дмитрием Карловским. Официальный слоган проекта – «fastest reactive micro-modular… framework»【6†L662-L670】. Он полностью написан на TypeScript и ориентирован на **малые быстрые приложения** с минимальными зависимостями. Главная идея $mol – предоставить **единую, согласованную архитектуру** «из коробки» (zero-configuration), чтобы не собирать фреймворк из десятков сторонних библиотек.

- **Ноль конфигов:** после клонирования стартового шаблона MAM и установки зависимостей приложение запускается без дополнительной настройки: `npm install && npm start`【6†L699-L704】【6†L662-L670】. Не нужно вручную настраивать сборщики, бандлеры или линтеры.
- **Микро-модули:** каждая функциональная часть ($mol называет их *модулями*) очень маленькая и опциональная【6†L669-L670】. При компиляции автоматически включаются только те модули, которые реально используются – зависимости подтягиваются по факту упоминания полного имени (Fully Qualified Name), без явных `import`【3†L963-L970】. Например, если вы в коде ссылаетесь на `$mol_state_local`, конструктор сам добавит соответствующий модуль в бандл【3†L963-L970】. Это избавляет от множества импорта/провайдеров и гарантирует «чистый» итоговый код.
- **Полная статическая типизация:** весь код пишется на TypeScript с отключённым `implicit any`【35†L13-L18】. Это обеспечивает надёжную разработку с подсказками IDE и предотвращает немедийные ошибки.

# Ключевые особенности $mol

$mol объединяет в себе сразу несколько возможностей, которые в других фреймворках требуют подключения отдельных библиотек. В официальном README перечислены основные преимущества: нулевая конфигурация, ленивый рендеринг, сквозная реактивность, авто-мемоизация и пр.【6†L662-L670】【6†L674-L683】. Стоит выделить особенно важные:

- **Реактивность через Data-Flow:** $mol реализует _архитектуру data-flow_, а не привычный control-flow. Все компоненты описываются как классы с реактивными свойствами. Каждое поле (свойство) класса считается функцией от других свойств. Система автоматически отслеживает зависимости: при изменении входных данных пересчитываются только зависящие свойства【3†L1023-L1030】. Это решает проблемы утечек памяти и каскадного обновления: ненужные объекты уничтожаются автоматически【3†L1034-L1041】. Благодаря этому не нужно вручную управлять подписками/отписками – реактивность «работает сама».
- **Кеширование и асинхронность «под капотом»:** $mol умеет кэшировать вычисления. Если метод помечен как реактивный (через декоратор `@$mol_mem`), то его результат запоминается, а повторные вызовы возвращают кеш【8†L1133-L1138】【6†L679-L683】. При изменении зависимостей кеш инвалидируется автоматически. Для асинхронных операций используется специальный механизм: если свойство не может сразу вернуть значение, оно бросает исключение `$mol_atom_wait`, а при поступлении данных система автоматически достроит интерфейс. Практически это означает, что код запросов к API можно писать как синхронный, а UI обновится «само по себе» при получении данных【3†L1042-L1047】【6†L679-L683】.
- **Ленивый рендеринг и виртуализация:** визуальные компоненты в $mol всегда ленивые. Базовый класс `$mol_view` гарантирует, что содержимое рендерится только при необходимости【3†L1005-L1012】. Существует встроенный механизм виртуального списка (`$mol_list` и `$mol_scroll`), который автоматически измеряет видимую область и **не рендерит невидимые элементы**. Благодаря этому интерфейсы остаются отзывчивыми даже при большом объёме данных. Как показывают бенчмарки, $mol без проблем «не рисует» DOM за пределами экрана, чего обычно нет ни в React, ни в Vue【10†L72-L74】【10†L94-L96】.
- **Офлайн и локальное хранилище «из коробки»:** $mol поддерживает концепцию *local-first*. Для работы со стейтом предоставлены встроенные модули: `$mol_state_local`(привязан к localStorage),`$mol_state_session`, `$mol_state_history` и т.д.【8†L1133-L1138】. Это позволяет сразу сохранять данные приложения локально без сложной настройки. Документация подчёркивает, что приложения на $mol «работают в офлайне и могут быть установлены как PWA»【16†L786-L794】.
- **Интернационализация (i18n):** фреймворк автоматически собирает строки локализации. При сборке генерируется файл вида `web.locale=en.json`, куда экспортируются все текстовые ресурсы из шаблонов и файлов `*.locale=en.json`【30†L1-L4】【16†L786-L794】. Это позволяет легко поддерживать несколько языков без сторонних библиотек: достаточно завести нужные JSON-файлы.
- **Небольшой объём и высокая скорость:** по замыслу автора $mol-приложение весит очень мало. Демо-приложение со 13 виджетами (виртуальный калькулятор со сложными формулами) без минификации заняло лишь **24 КБ**【13†L91-L98】, что меньше даже чистого React. В реальных проектах бандл часто не превышает 100 КБ **всего кода** (не сжатого). Сам Дмитрий Карловский отмечает в твиттере: «У нас бандлы меньше 100КБ. И код даже не минифицирован. Причины тут: минимум бойлерплейта, максимум переиспользования, грамотная архитектура»【39†L33-L36】. За счёт микро-модульности и tree-shaking на основе упоминаний неиспользуемый код в бандл просто не попадает【3†L963-L970】. В результате приложение стартует почти мгновенно (заявлено «<¼ секунды»)【16†L786-L794】.
- **Автоматическая BEM-генерация стилей:** $mol-элементы генерируют свои селекторы на основе имени компонента, обеспечивая семантичные и уникальные классы. Например, при описании компонента создаётся атрибут вида `my_app_sidebar_body`, и CSS-правила пишутся через этот селектор【34†L761-L764】. Благодаря этому селекторы человечески читаемы и **никаких рандомных хешей в классах нет**【34†L761-L764】.

# Архитектурные паттерны $mol

- **Классы вместо функций:** весь UI в $mol строится через классы и декларативные шаблоны (view.tree, view.ts, view.css). Разработчик описывает компонент как класс `$MyComponent extends $.$MyComponent`, а затем переопределяет методы и свойства. Любой метод в таком классе – это потенциальная точка расширения: его можно заменить, расширить или удалить. Благодаря этому **наследование – норма**, а не исключение, что сильно упрощает переиспользование кода【3†L982-L991】.
- **Без «магии» сторонних библиотек:** во многих современных стэках (React+Redux/Context/React Query/MobX/и т.д.) приходится вручную соединять роутеры, стейт-менеджеры, кеши запросов, адаптивные решения, локализацию и др. В $mol все эти задачи решены в рамках единого фреймворка: нет нужды устанавливать react-query для запросов (есть `$mol_http`/`$mol_fetch`), нет нужды придумывать кеширование (реактивные свойства и local state), нет нужды подбирать виртуализацию (есть встроенная), нет нужды придумывать i18n (есть механизм `.locale.json`). Как описывает автор, типичный React-разработчик всегда «пилит костыли» (numerous dependencies), а в $mol ничего подобного не требуется【34†L759-L764】【16†L786-L794】.
- **Единая система компонентов и модулей:** в $mol компонент **равнозначен приложению и модулю**. Нет отдельного JSX/рендера: один механизм описания работает и для микровиджета, и для целой страницы. При этом соблюдается строгая конвенция: пространство имён соответствует папкам проекта. Компонент `$MyApp`обязан лежать в папке`my/app` и иметь такое же имя – иначе фреймворк его не найдёт. Это жёсткое правило предотвращает хаос в больших кодовых базах (раз у класса уникальный full path, он гарантированно однозначно связывается с данным компонентом).
- **Отладка и прозрачность:** $mol даёт подробные человекочитаемые идентификаторы всем объектам и DOM-узлам【8†L1085-L1092】. В консоли браузера можно быстро получить любой компонент по его id. Исключения обрабатываются особым образом, чтобы легко понять, где произошла ошибка (вместо нагромождений `try/catch`). Всё это упрощает поддержку и дебаг сложных приложений.

# Сравнение с React/Vue и другими

**React/Vue** – популярные, но при масштабировании часто требуют большого числа библиотек и шаблонов. Типичные «боли» привычного стека и ответы $mol:

- _Боль:_ для простых вещей (например, офлайн-кэш, локализация, виртуальный скролл) нужно ставить и настраивать отдельные пакеты.  
  _$mol:_ всё перечисленное есть «из коробки»【16†L786-L794】. Офлайн-режим и локальное хранилище обеспечивается `$mol_state_*`, адаптивность через CSS-возможности фреймворка, виртуализация – через `$mol_list` и `$mol_scroll`【3†L1005-L1012】【10†L72-L74】. Нет необходимости писать свой код на основе сторонних SDK.

- _Боль:_ в React/Vue много «обвязки»: хуки `useEffect`, промисы, ручные подписки. Сложно догадаться, когда уйдёт лишний ререндер, и нужно много мемоизации/тюннинга.  
  _$mol:_ тут другая модель. Не надо вставлять эффекты – реактивные свойства сами обновляют UI. (Например, $mol_wire позволяет писать асинхронно “псевдосинхронный” код без явных `async/await`【6†L679-L683】.) Перерисовывается только то, что зависит от изменившихся данных, без внезапных лишних обновлений.

- _Боль:_ большое количество npm-зависимостей (библиотек), которые постоянно обновлять, ломают билды и увеличивают вектор атак.  
  _$mol:_ сдержанный набор пакетов. Фреймворк живёт в едином экосистемном пространстве MAM, все модули своего проекта – это ваши директории. Никаких сторонних UI-библиотек и тяжёлых сборок – только собственный код (tree-shaking по полным именам【3†L963-L970】). Это уменьшает риски supply-chain и упрощает сопровождение.

- _Боль:_ проблемы с SSR и гидротацией – различный рендер на сервере и клиенте, баги с согласованием состояний.  
  _$mol:_ подход «тот же код и на dev, и на prod»【39†L67-L70】 и единый реактивный механизм позволяют упростить SSR. Хотя $mol и поддерживает разные платформы (Node, браузер, Cordova【6†L674-L680】), основная идея – как можно меньше «различий» между окружениями.

- _Боль:_ большие проекты с централизованным стейтом (Redux, MobX) сложно масштабировать и отлаживать.  
  _$mol:_ не требует внешних стор-менеджеров. Данные лежат в реактивных свойствах компонентов и `$mol_state_*`. Все зависимости очевидны в коде классов – поэтому граф состояний ясен по коду, а не скрывается за `dispatch`/`selector`. При этом автоматически есть возможность группировать обновления («batch update») – реактивная архитектура оптимизирует пересчёты свойств.

В целом $mol позиционирует себя как решение, устраняющее системные противоречия фронтенд-разработки: меньше «хаков», больше предсказуемости. Как отмечал автор на Хабре, рекламированные технологии (React/Redux) часто диктуют разрабатывать одни и те же компоненты заново из-за жёстких шаблонов и изоляции【34†L759-L764】. $mol снимает эти ограничения – в нём не бывает «невозможных» изменений: любой компонент можно расширить, переопределить стиль, заменить часть логики.

# Примеры и внедрение

Несмотря на малую известность, у $mol есть реальные проекты. Например, сайт митапа PiterJS (piterjs.org) сделан на $mol: регистрация на событие там в один клик (регистрация идет через внешнее API) и получается очень лёгкий фронт без сложной обвязки. Портфолио проектов $mol можно посмотреть на выставке приложениях (showcase.hyoo.ru).

В блоге автора подробно описан пример – приложение для заметок, которое было реализовано **за 2 часа**【34†L776-L784】. Оно сразу поддерживало офлайн-режим, поиск с подсветкой, работу в локальном хранилище и другие нужные фичи【34†L781-L789】. Другая демонстрация – электронная таблица с формулами и виртуальным скроллом – показала загрузку только 24 КБ и молниеносный отклик даже на слабом устройстве【13†L91-L99】.

# Заключение

Для CTO и команды важны **стабильность** и **скорость разработки**. $mol предлагает строгую архитектуру и готовый набор средств, чтобы _не придумывать велосипед_ при каждом проекте:

- **Быстрый старт:** нет мучений с конфигами – скачать MAM-репозиторий и запустить dev-сервер. Структура папок чётко определена, что упрощает онбординг новых разработчиков.
- **Минимум зависимостей:** меньше «npm-боли» и supply-chain проблем. Фреймворк развивается самим сообществом $mol/MAM и обновляется разом, а не через сотни сторонних пакетов.
- **Полная реактивность:** бизнес-логика и UI описываются единообразно, нет «скрытых» слоёв. Все свойства/методы можно легко переопределить или протестировать – это даёт гибкость масштабирования системы.
- **Высокая производительность:** модульность и ленивость позволяют держать интерфейс быстрым даже на больших данных. Показано, что $mol-приложение рендерит только видимую часть, без лишней нагрузки【10†L72-L74】.
- **Встроенные best practices:** offline-first, i18n, адаптивность и другие фичи – уже учтены. Не надо в последний момент доделывать, всё изначально поддерживается.

Таким образом, $mol – не просто еще один фреймворк. Это попытка взять большинство «хотелок» для современных веб-приложений и заложить их в основу. Если разработчики устанут от бесконечной настройки React/Vue-стека и поиска «лучших» библиотек, $mol может быть интересной альтернативой. Он заточен под долгосрочное сопровождение крупных приложений с множеством бизнес-экранами (формы, таблицы, админки). Его подход **меньше зависит от трендов** – обратный концептуальный ход по сравнению с быстро изменяющейся экосистемой JS【6†L674-L680】【34†L759-L764】.

**Источники:** документация и обсуждения $mol【3†L963-L970】【6†L662-L670】【34†L761-L764】【16†L786-L794】, статьи и блоги Дмитрия Карловского【10†L72-L74】【39†L33-L36】, а также материалы по сравнению с React/стратегиям фронтенда. Все цитаты даны в формате «【линия】» из открытых ресурсов.
