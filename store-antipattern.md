# Вам не нужен стор, стор это антипаттерн

Здравствуйте, товарищи! Сегодня у нас будет статья про state management во фронтенде, с непривычного угла. Обычно про это пишут уставшие React-разрабы, которые семь лет тащили на себе Redux и наконец сорвались(точно об этом обычно пишут утсавшие реакт разрабы то ? докажи или убери ссылками докажи). Я не такой. Я пишу на $mol второй год, и за это время я **ни разу не написал ни одного редьюсера**. Расскажу, как я к этому пришёл, что вижу со стороны и почему вам, особенно если вы только заходите во фронтенд, **не надо учить Redux в принципе**.

В первой части короткий рассказ обо мне и моём пути. Во второй — почему стор это антипаттерн, со скриншотами чужих страданий. В третьей — как живёт без стора моё приложение, на куске реального кода. В четвёртой — типовые возражения. Поехали. (ии слоп абзац, перечитай статьи валика, перепиши)

## Стор глазами стороннего наблюдателя

У меня есть чатик с фронтами, в основном React/Angular/Vue. ( тут был вообще бред выдуманный, я там активно пишу  в чате вообще то , но не упоминай об этом) 

Вот выдержки за последние пару месяцев, цитирую: 

> — Мы используем редакс, у нас стор хранит в себе все данные приложения, и при запуске devtools он падает, потому что не смог через девтулзы пропустить всё то говно, которое они туда накидали.

> — Я просто считаю, что нехуй тянуть в глобал стор каждую хуйню и превращать его в советский балкон.

> — Я не могу себя заставить на интеграционные перейти, так как мокать кучу рестов, кучу изменений стора — короче, пипец.

> — Из-за того что у нас легаси-чудовище, а сейчас ещё и возобновили поддержку версии 5-летней давности, то я в ахуе.

«Советский балкон» — ( ТИРЕ!!!!!!!!!!!!) это, по-моему, самая точная характеристика того, во что превращается типичный Redux-стор в реальном продакшене. Если вы никогда не видели советский балкон ( сука все видели, все тут из снг блять ) — это место, куда сваливается всё, что вроде бы нужно, но непонятно где должно лежать. 

Так и стор. Когда я смотрю на типовой Redux-стейт в больших проектах, я вижу:

```ts
{
    auth:     { ... },
    entities: {
        posts:    { byId, allIds },
        users:    { byId, allIds },
        comments: { byId, allIds },
    },
    ui: {
        sidebar:   { open: false },
        modal:     { type: null, payload: null },
        forms:     { register: { email: '', password: '' } },
        hover:     { itemId: null },         // ← hover, серьёзно
    },
    pagination: { posts: { page: 1, total: 50 } },
    cache:      { ... },                      // ← RTK Query
}
```

Hover лежит в глобальном сторе рядом с access-токеном пользователя. Состояние формы регистрации тоже там. По принципу советского балкона — «вроде нужно, не выкидывать же, положу сюда».( опять тире блят, заебал ты заставляешь повторяться )

Меня в чате на этот счёт пытались переубедить:

> — Касательно оптимизации это спорный вопрос. Не понимаю, каким образом вес говна в сторе влияет на оптимизацию.

И знаете, формально он прав — в Redux-архитектуре вес стора не сильно влияет на скорость. Это его, в общем-то, единственный плюс — он не тормозит. Но оптимизация не должна быть оправданием для архитектурного бардака.

## Что меня удивило, когда я разобрался как устроен React

Я довольно долго не мог понять, **зачем вообще нужен стор как отдельная сущность**. Мне казалось что везде компонент сам может хранить своё состояние. Я же это вижу в $mol каждый день. Положил поле в класс, повесил `@$mol_mem`, всё работает.

Был на фронтенд подлодке, смотрел лекцию про реакт, возник вопросец:

> — В реакте нужно вручную вызывать функцию, чтобы отрендерить ответ с бэка?

Ответ был — да, в общем-то, нужно. То есть в React **компонент сам по себе не знает**, что данные изменились. Ему надо это сказать через `setState`, или через подписку на стор, или через хук типа `useQuery`. Если вы записали значение в обычное поле объекта — React этого не заметит, ререндера не будет(тире блять).

И вот тут до меня дошло, **зачем нужен Redux**. Он нужен потому, что **в React нет встроенной реактивности**. Стор — это внешняя система слежения за изменениями, потому что сам React за изменениями не следит. Это не «архитектурный паттерн», это **костыль для отсутствующей фичи языка/фреймворка**. Беря реакт вы собираете свой костыльный фреймворк, получается

Vue в этом смысле умнее: там есть `ref` и `reactive`, которые слегка прикручены прямо к фреймворку. Но Vue-комьюнити всё равно к этому добавило Vuex, потом Pinia — те же сторы, только сбоку. Зачем ?

В $mol всё иначе. Каждое поле, обёрнутое в `@$mol_mem`, **автоматически** уведомляет компоненты, которые его читают. Изменил — (тире блять) DOM обновился. Без подписок, без диспатчей, без хуков. Это и есть «реактивность из коробки» в чистом виде.

## Стор размножается

Вот ещё цитата, которая меня поразила:

> — Я только щас понял что [коллега] написал) data из квери-хука не будет в сторе храниться, пока ты туда не засунешь вручную.

Стоп. То есть у вас есть Redux-стор, есть `useQuery` из TanStack, и **они не связаны между собой**. Запрос сходил на бэк, данные пришли, кэш TanStack-а их сохранил, но в Redux они **не попадут**, пока вы их туда руками не положите. И в Redux-комьюнити это норма.

Я уточнил у того же знакомого:

> — Сейчас стандарт кэш хранить в редакс, так как 99% юзают RTK query или react query.

То есть **сам Redux уже не справляется** — все используют RTK Query или TanStack Query, потому что в чистом Redux работать с асинхронкой больно. И теперь у вас:

1. **Redux** — для глобального состояния
2. **RTK Query / TanStack Query** — для серверного кэша
3. **useState** — для локального компонентного состояния
4. **useReducer** — для среднего, между локальным и глобальным
5. **Context** — для прокидывания мимо props

Пять( нахуя ты шестое то блять написал, никто в здавом уме два стора не будет использовать) подсистем хранения состояния. В одном приложении. Каждая со своими правилами, API, паттернами. И разработчик каждый раз решает, **куда** положить очередную переменную.

У меня в $mol — одна подсистема(тире и далее короткие предложения блять, замени точки на запятые, я так больше люблю, все комменты отностятся ко всему тексту!!!). Реактивные поля компонента. Всё. Локальное лежит в компоненте, общее поднимается в общего предка, глобальное — в синглтоне App. Где живёт состояние — диктует **дерево компонентов**, а не выбор библиотеки.

## Как это устроено в моём приложении

Недавно я  запилил проект под названием [Blitz](https://github.com/hyoo-ru/mam/tree/master/bog/blitz) — это что-то вроде Kahoot, командный квиз в реальном времени. Хост заводит сессию, игроки заходят по коду, отвечают на вопросы, видят баллы в реальном времени. Без бэкенда — всё через Giper-CRDT-базу [Giper Baza](https://github.com/hyoo-ru/mam/tree/master/giper/baza), которая сама синкает данные между устройствами.

Сам проект — мой магнум опус, там реализовано всё: блог, профили, лобби, мультиплеер, реакции. По хорошему — поле для стора огромное. В React-варианте этого приложения был бы солидный slice на сессию, slice на игроков, slice на текущий вопрос, slice на UI, slice на профиль, плюс RTK Query на серверные данные. Я думаю, файлов было бы под сотню.

У меня сторов нет.

Вот, например, как описан игрок:

```ts
export class $bog_blitz_player extends $giper_baza_dict.with({
    Score:       $giper_baza_atom_real,
    Name:        $giper_baza_atom_text,
    IsHost:      $giper_baza_atom_bool,
    Avatar:      $giper_baza_atom_link_to( () => $giper_baza_file ),
    Answer_land: $giper_baza_atom_text,
}) {}
```

Это **схема данных**. Я говорю — игрок состоит из счёта, имени, флага «хост ли он», аватарки и ссылки на отдельный land с его ответами. Всё. Никаких actions, reducers, selectors. Просто описание полей.

В коде я с ним работаю так:

```ts
const player = $bog_blitz_player.of( player_id )

player.Name()              // ← чтение имени
player.Score( 42 )         // ← запись счёта
player.IsHost()            // ← реактивная подписка на флаг
```

Хост на одном устройстве пишет `player.Score( 42 )`. Через секунду на устройстве игрока обновляется счёт **сам**, потому что Giper Baza засинкала изменение, а $mol узнал что данные поменялись и перерендерил view. Никакого диспатча, никакого `socket.on('score-update', ...)`, никакой ручной синхронизации.

А вот так у меня устроен переход между экранами в самом приложении:

```ts
screen( next?: string ) {
    if( next !== undefined ) {
        this.mobile_menu_showed( false )
        if( next === 'lobby' ) {
            this.$.$mol_state_arg.value( 'quiz', null )
        }
    }
    return this.$.$mol_state_arg.value( 'screen', next || undefined ) || 'admin'
}
```

`$mol_state_arg` — это двусторонняя связь поля с query-параметром URL. То есть **текущий экран приложения лежит прямо в адресной строке**. Открыл `?screen=lobby` — приложение на лобби. Нажал кнопку «в админку» — URL поменялся на `?screen=admin`. Бесплатные deeplink-и, бесплатный shareable state, бесплатная история навигации через history.back. И никакого React Router + Redux + sync-middleware.

Что у меня лежит локально в самих компонентах:

- открыто ли мобильное меню
- какой пункт в навигации сейчас наведён
- содержимое формы создания квиза до отправки
- развёрнут ли блок с правилами

Всё это **не существует** на уровне общего стора. Оно живёт там, где используется. А когда компонент уничтожается — оно само себя удаляет.

Когда я объяснял эту схему React-разрабу, он на полминуты завис и сказал: «Подожди, а где у тебя стор?». Я говорю: его нет. Он спросил: «А как ты тогда шаришь данные между экранами?». Я ему показал то же `$mol_state_arg`. Он сказал: «Ну это же костыль, использовать URL как глобальное состояние». Я ответил, что URL — это и есть **самый надёжный** глобальный стор, который встроен в браузер с 1993 года, бесплатно, с поддержкой forward/back, поделиться-кнопкой, букмарками и индексацией в Google. Костыль, по-моему, наоборот — это **не использовать URL** и тащить параллельный Redux-стор сбоку.( ии слоп придумка, прям как слоп читается но мысль правильная, напиши ее от моего имени, не в форме рассказа)

## Возражения

Я уже два года показываю $mol-код знакомым. Основная претензия — **синтаксис `view.tree`**.

> — Какой view.tree ужасный.

> — Это вообще не похоже на JSX.

> — Я не хочу учить новый DSL.

Я отвечаю одинаково: **дело привычки, не более**. View.tree — это структурированный DSL, который декларативно описывает дерево компонентов, это абстракция над html, которая транспилируется в обычный js. JSX — это HTML, прикрученный к JavaScript со всеми его недостатками.

Через две недели работы с view.tree вы перестаёте его замечать. Через месяц вы понимаете, что он **строже** JSX-а — нельзя случайно засунуть в дерево что-то лишнее, нельзя написать «логику в шаблоне», нельзя сделать typo в имени компонента (его проверит компилятор tree). Это тот же контраст, что между TypeScript и JavaScript: первое строже, второе свободнее, и через месяц TypeScript-а вы не хотите обратно.

Остальные возражения:

— **«А что с SSR?»** — для статичног prerender'a я написал github action, для динамического нужно делать что то своё 
— **«Подходит ли оно для командной разработки ?»** — Более чем! Даже лучше всех остальных! Иерархия owner'ов в $mol жёстче любого Redux-style-guide, новый человек физически не может «случайно положить state не туда».
— **«А найм?»** — берёте любого TypeScript-разраба, который понимает классы и реактивность. Онбординг 2-4 недели. Это меньше, чем разобраться в чужой React-кодбазе с её Redux/RTK/Zustand/TanStack-зоопарком.

## Что я хочу сказать новичкам

Если вы только заходите во фронтенд — у меня к вам один совет.

**Не учите Redux.** Не учите Zustand, Pinia, Jotai, MobX, RTK Query, TanStack Query. Не из принципа, а потому что если вы возьмёте $mol — вам **не понадобится ничего из этого**. Вы не будете писать редьюсеры, потому что компонент сам себе редьюсер. Вы не будете писать селекторы, потому что геттер компонента — это и есть селектор. Вы не будете писать async middleware, потому что `$mol_fetch` работает синхронным кодом. Вы сэкономите примерно полгода обучения и потом будете писать в три раза меньше кода.

Когда я пришёл во фронтенд и попал на $mol — я **сразу начал с хорошего**. Не научился плохому. И когда мои знакомые в чате жалуются на то, что у них «советский балкон» в Redux-сторе, на упавшие devtools, на муторные интеграционные тесты с моками — я постоянно невпопад умпоминаю мол и всех разражаю :) ( уже нет ), потому что у меня этого нет. Не я молодец. У меня просто фреймворк такой.

Если вы — бывалый React-разраб, и вы дочитали до этой строки — спасибо. Я не буду вас уговаривать переучиваться, потому что у вас есть проекты, команда, инерция, найм и прочая бытовая правда. Но **на pet-проект** возьмите $mol на выходных. Посмотрите, как это, когда стор не нужен. Может, понравится.

А если вы новичок, и фронтенд для вас ещё чистый лист — то всё ещё проще. Идите на [mol.hyoo.ru](https://mol.hyoo.ru), почитайте статьи Карловского, посмотрите выступления на PiterJS. И начните **сразу с хорошего**, как когда-то начал я.

Спасибо за внимание, товарищи. Вопросы и возражения — в комментарии.
