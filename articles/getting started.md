## $mol Getting started

**Приложения, которые работают офлайн, синхронизируются без бэкенда, который надо писать, и остаются крошечными.** Реактивность, хранилище и UI — в одном стеке. Ты пишешь приложение, а не собираешь фреймворк из библиотек.

К концу этого руководства у тебя будет настоящее приложение на $mol — с офлайном, темами и локальным хранилищем, развёрнутое локально и собранное одной командой. Минут за десять.

## Что получится в итоге

Запущенное приложение, в котором уже есть то, что обычно прикручиваешь потом:

- установка как PWA и офлайн-режим,
- локальное хранилище с бесконфликтной синхронизацией (без серверного кода с твоей стороны),
- светлая и тёмная темы с переключателем,
- клиентский роутинг,
- GitHub Action для деплоя на Pages.

Всё это даёт скаффолдер — а дальше ты поменяешь одну строку и увидишь, как оно само пересчитывается.

## Что понадобится

- [Node.js](https://nodejs.org/) (свежий LTS) и `git`.

Это весь список. $mol включает в сборку только те модули, на которые ссылается твой код — никакого списка зависимостей, который надо вести руками.

## Шаг 1 — Поднять рабочее пространство и dev-сервер

```bash
git clone https://github.com/hyoo-ru/mam.git ./mam && cd mam
npm install && npm start
```

Dev-сервер теперь работает на `http://localhost:9080`. Он собирает каждый бандл по запросу: запрашиваешь страницу — собирается JS и CSS ровно для этого модуля и его зависимостей, ничего лишнего.

## Шаг 2 — Сгенерировать приложение (одна команда)

Из папки `mam` создай проект. Выбери свой неймспейс и имя приложения; здесь — `my/hello`:

```bash
npm create view-tree-lsp@latest my/hello -- --no-docker --no-tauri
```

Команда создаёт готовое работающее приложение в `./my/hello/`:

```
my/hello/
├── app/
│   ├── index.html        # веб-точка входа
│   ├── app.view.tree     # декларативное описание компонента
│   ├── app.view.ts       # поведение
│   ├── app.view.css.ts   # стили (типизированный CSS-in-TS)
│   ├── app.test.ts       # тест
│   ├── app.meta.tree     # офлайн установка 
│   └── app.locale=ru.json
├── store/
│   └── store.ts          # локальное хранилище
├── assets/logo.svg
└── .github/workflows/deploy.yml
```

Приложение готово к запуску.

## Шаг 3 — Открыть

Перейди на **`http://localhost:9080/my/hello/app/`**. Увидишь приложение с темами, несколькими экранами и работающей навигацией. Его слой данных уже настроен на офлайн-first и бесконфликтную синхронизацию — без серверной логики с твоей стороны.

> Скаффолдер также печатает URL тест-раннера (`…/app/-/test.html`) — там можно посмотреть, как проходит встроенный тест.

## Что ты получил (и почему это важно)

Дефолтное приложение — не пустой «hello world», а минимальное *настоящее* приложение на $mol, и это сделано намеренно. То, на что обычно уходит спринт, уже связано:

- **Локальное хранилище из коробки.** `store/store.ts` описывает хранилище поверх бесконфликтного реплицируемого типа (CRDT). Состояние сохраняется локально и чисто сливается между устройствами — тебе не нужно писать логику синхронизации или бэкенд.
- **Офлайн это просто еще один модуль** В `app.meta.tree` подключён модуль офлайн-установки; приложение ставится как PWA и продолжает работать без сети.
- **Темы и роутинг уже есть.** Переключатель светлой/тёмной темы и навигация по URL (`$mol_state_arg`) идут вместе со скаффолдом.
- **Типизированные стили.** Стили живут в `app.view.css.ts` как типизированный CSS-in-TS — опечатка становится ошибкой компиляции, а не молчаливым «ничего не произошло».

## Сделать реактивно

Открой `app/app.view.tree`, найди приветственный текст. Добавь поле ввода, привязанное к свойству, и строку, которая от него зависит:

```tree
home <= Home $mol_page
	title \Home
	body /
		<= Name $mol_string
			hint \Имя
			value? <=> name? \
		<= Greeting $mol_text
			text <= greeting \
```

Затем в `app/app.view.ts` опиши `greeting` как функцию от `name`:

```typescript
greeting() {
	const name = this.name()
	return name ? `Привет, ${name}!` : ''
}
```

Обнови и печатай. Приветствие обновляется само. Ты не подписывался на поле и не планировал перерисовку: `greeting` читает `this.name()`, поэтому $mol запоминает зависимость и пересчитывает только это значение, когда меняется `name`. Та же реактивная модель пронизывает все слои — включая загрузку данных, где значение, которое ещё не готово, «приостанавливается» и подставляется, когда придёт. Поэтому асинхронный код читается как обычный синхронный.

## Настроить редактор (минута)

`view.tree` чувствителен к отступам, так что поддержка в редакторе сильно помогает:

- **VS Code** — поставь [плагин языка view.tree](https://marketplace.visualstudio.com/items?itemName=valikov.tree-language-service); используй плагин `.editorconfig`, чтобы были **TAB-ы для отступов** и **LF в концах строк**.
- **Zed** — [расширение view.tree](https://zed.dev/extensions/viewtree) добавляет подсветку, go-to-definition и автодополнение.
-  Zed расширение  построено на [view.tree LSP](https://www.npmjs.com/package/view-tree-lsp) и [tree-sitter-грамматике](https://github.com/Dev-cmyser/tree-sitter-viewtree) — их можно подключить и к другим редакторам.

## Деплой

В скаффолде уже лежит GitHub Actions workflow: пушишь в `master` — он собирает и публикует на GitHub Pages. Ветки-фичи деплоятся на собственный preview-URL.

## Попробуй

Сгенерируй приложение, преврати приветственный экран во что-то, чем правда пользуешься, и положи это в сгенерированное хранилище — чтобы пережило обновление страницы и перезапуск офлайн. Когда понадобится полный список кирпичиков — поля, списки, графики, пикеры — в [справочнике модулей](https://github.com/hyoo-ru/mam_mol) есть каждый, с примерами.
